Q. 스패처 서블릿(DispatcherServlet)의 장단점:
```
장점:
중앙 집중적인 요청 처리: 디스패처 서블릿은 웹 애플리케이션에서 클라이언트의 모든 요청을 중앙에서 처리하는 역할을 수행하여 일관된 처리 로직을 구현할 수 있습니다.
유연한 핸들러 매핑: 디스패처 서블릿은 핸들러 매핑을 통해 요청과 처리할 핸들러(컨트롤러)를 매핑하므로, 다양한 핸들러 매핑 전략을 적용할 수 있습니다.
공통 기능의 모듈화: 인터셉터를 사용하여 공통적인 기능(로그 기록, 인증, 권한 검사 등)을 모듈화하고 재사용할 수 있습니다.
유연한 뷰 처리: 디스패처 서블릿은 뷰 리졸버를 통해 요청 결과를 적절한 뷰에 전달하여 렌더링할 수 있습니다.
단점:
추가적인 구성 요소 필요: 디스패처 서블릿을 사용하기 위해서는 핸들러 매핑, 핸들러 어댑터, 뷰 리졸버 등 추가적인 구성 요소를 설정해야 합니다.
약간의 오버헤드: 디스패처 서블릿은 요청의 처리를 위해 중간 단계를 거치므로 약간의 오버헤드가 발생할 수 있습니다.
```

Q. 요청이 들어왔을 때 핸들러를 거쳐서 컨트롤러로 위임하는 이유는 다음과 같습니다:
```
요청과 해당 요청을 처리할 컨트롤러를 매핑하기 위해: 
클라이언트의 요청은 다양한 URL 패턴으로 들어올 수 있습니다. 핸들러 매핑을 통해 요청과 처리할 컨트롤러를 매핑하고, 이를 통해 요청을 적절한 컨트롤러에 위임합니다.

컨트롤러에서 실제 비즈니스 로직을 수행하기 위해: 
컨트롤러는 클라이언트의 요청을 처리하고 필요한 비즈니스 로직을 수행합니다. 따라서 핸들러를 통해 컨트롤러로 위임하여 비즈니스 로직을 실행하고 응답을 생성합니다.

공통적인 기능의 처리를 위해: 
핸들러를 거쳐서 컨트롤러로 위임하는 과정에서 인터셉터를 사용하여 공통적인 기능(인증, 로깅, 권한 검사 등)을 처리할 수 있습니다.
```

Q. 핸들러 어댑터를 사용하는 이유
```
다양한 컨트롤러 타입을 지원하기 위해: 
스프링은 다양한 컨트롤러 타입을 지원합니다. 핸들러 어댑터는 이러한 다양한 컨트롤러 타입에 대해 표준화된 인터페이스를 제공하여 일관된 방식으로 핸들러를 실행할 수 있도록 도와줍니다.

컨트롤러의 실행을 캡슐화하기 위해: 
핸들러 어댑터는 컨트롤러의 실행을 캡슐화하여 DispatcherServlet이 핸들러를 실행하는 과정을 단순화하고 표준화합니다. 핸들러의 실행 결과를 처리하고 적절한 뷰를 결정하는 역할을 수행합니다.

유연한 확장성을 제공하기 위해: 
핸들러 어댑터는 확장 가능한 구성 요소로써 개발자가 커스텀 핸들러 어댑터를 구현하여 특정한 컨트롤러 타입에 대한 처리 방식을 변경하거나 확장할 수 있도록 합니다.
```

Q. 프론트 컨트롤러(Front Controller)가 필요한 이유
```
중앙 집중적인 요청 처리: 
프론트 컨트롤러는 웹 애플리케이션에서 클라이언트의 모든 요청을 중앙에서 처리하는 역할을 수행합니다. 모든 요청을 한 곳에서 처리함으로써 일관된 처리 로직을 구현하고 중복 코드를 줄일 수 있습니다.

공통 기능의 모듈화:
프론트 컨트롤러는 인터셉터를 사용하여 공통적인 기능(인증, 로깅, 권한 검사 등)을 모듈화하고 재사용할 수 있습니다. 이를 통해 코드 중복을 줄이고 유지보수성을 향상시킬 수 있습니다.
```
